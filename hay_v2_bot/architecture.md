# Архитектура: Telegram-бот Hay V2

## Обзор

`hay_v2_bot` — это модульный Telegram-ассистент, построенный вокруг пайплайнов Haystack, векторного хранилища Pinecone и ingestion-документов через Docling.

Ключевые цели:
- сохранить все возможности бота v1;
- добавить обработку файлов и retrieval с учетом документов;
- сделать архитектуру расширяемой и удобной для поддержки.

## Компоненты верхнего уровня

- `main.py`
  - точка входа приложения;
  - загрузка env/конфигурации;
  - инициализация сервисов и пайплайнов;
  - запуск Telegram polling.

- `bot/app.py`
  - обработчики Telegram (`/start`, `/help`, `/memory`, `/dogfact`, `/dogphoto`, `text`, `document`);
  - оркестрация пайплайнов и компонентов;
  - отправка сообщений пользователю и обработка ошибок.

- `components/`
  - `memory_service.py`:
    - обертка над `PineconeManager`;
    - чтение/сохранение пользовательской памяти;
    - сохранение ответов ассистента;
    - методы записи чанков документов.
  - `docling_processor.py`:
    - проверка поддерживаемых форматов;
    - преобразование файлов в структурированные чанки через Docling;
    - резервное chunking-разбиение из markdown-экспорта.
  - `tools.py`:
    - инструменты с фактами о собаках и фото/vision-анализом.
  - `summarizer.py`:
    - генерация однофразового резюме для загруженных файлов.
  - `logging_config.py`:
    - настройка JSON-логирования.

- `pipelines/`
  - `generation_pipeline.py`:
    - чат-пайплайн Haystack для генерации ответа с учетом:
      - контекста памяти,
      - контекста найденных документов,
      - контекста инструментов.
  - `ingestion_pipeline.py`:
    - ingestion-пайплайн Haystack:
      - извлечение чанков через Docling;
      - запись подготовленных чанков в Pinecone.

## Хранилища данных и namespace

- индекс Pinecone:
  - хранит память и документы как векторные записи.

- namespace:
  - namespace по умолчанию: записи разговорной памяти;
  - `docs`: чанки документов для retrieval в RAG-сценарии.

Примеры метаданных записей:
- память:
  - `user_id`, `username`, `role` (`user` / `assistant`), `source`, `timestamp`, `text`.
- документы:
  - `filename`, `chunk_no`, `page_no` (если доступно), `text`.

## Рабочие потоки

### 1) Поток текстового сообщения

1. Telegram получает текст.
2. Приложение загружает память пользователя через `MemoryService`.
3. Приложение извлекает релевантные чанки документов из namespace `docs` в Pinecone.
4. При необходимости по интенту вызываются инструменты.
5. `generation_pipeline` собирает prompt и генерирует ответ.
6. Бот отправляет ответ пользователю.
7. Бот сохраняет:
   - сообщение пользователя (`role=user`);
   - ответ ассистента (`role=assistant`).

### 2) Поток файлового сообщения

1. Telegram получает файл (`document`).
2. Бот отправляет статус о начале обработки.
3. Файл скачивается во временный путь.
4. Запускается `ingestion_pipeline`:
   - конвертация и извлечение чанков через Docling;
   - запись чанков в Pinecone (namespace `docs`).
5. Бот отправляет сообщение о завершении.
6. Бот отправляет однофразовое резюме файла.
7. Сообщения ассистента сохраняются в Pinecone.

### 3) Поток команд

- `/start`, `/help`, `/memory`, `/dogfact`, `/dogphoto` обрабатываются в `bot/app.py`.
- все выходные сообщения ассистента сохраняются в Pinecone с `role=assistant`.

## Обоснование дизайна пайплайнов

- пайплайны Haystack дают явную графовую связку компонентов и упрощают замену модулей;
- ingestion и generation разделены по ответственности:
  - ingestion тяжелее и ориентирован на документы;
  - generation ориентирован на низкую задержку ответа;
- такое разделение позволяет в будущем вынести ingestion в async/background режим без изменения чат-потока.

## Стратегия обработки ошибок

- `try/except` на уровне обработчиков в Telegram-слое;
- понятные fallback-сообщения пользователю при сбоях;
- структурированные JSON-логи для диагностики;
- временные файлы удаляются в блоках `finally`.

## Точки расширения

- замена LLM-моделей через env (`OPENAI_CHAT_MODEL`, `OPENAI_VISION_MODEL`);
- изменение namespace для документов через `PINECONE_DOCS_NAMESPACE`;
- смена стратегии chunking в `docling_processor.py`;
- добавление reranker перед генерацией;
- добавление фильтрации документов по пользователю в retrieval-метаданных.

## Безопасность и эксплуатационные заметки

- секреты загружаются из переменных окружения;
- бинарные файлы не хранятся постоянно (только временно);
- расширение файла валидируется до ingestion;
- при росте объема памяти/документов нужно контролировать размер Pinecone и стратегию retention.
